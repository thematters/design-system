# Phase 0–6 Archive

How the Matters Design System repo went from "scattered Figma + Google
Drive + Notion" to a working monorepo with published packages, public
docs, visual regression, and an HTML→PNG template pipeline — over one
weekend in late April 2026.

This is a one-page narrative for someone seeing the repo for the first
time. For decision context per phase, read each phase's PR description
on GitHub. For the per-package release log, see
`packages/<name>/CHANGELOG.md` (auto-generated by changesets).

---

## Phase 0 — Bootstrap (PR #29)

## Phase 1 — Monorepo + Button (PR #30)

The repo grew from a flat tree into a **pnpm workspaces monorepo**:

- `packages/tokens/` — canonical token source moved here from the root.
  Build now writes to both `packages/tokens/dist/` (npm) and
  `tokens/dist/` (legacy vendored-copy back-compat) so existing
  consumers don't need to change a path.
- `packages/react/` — new home for React component implementations
  (`@matters/design-system-react`)
- `apps/storybook/` — Storybook 8 + Vite docs site
- `.github/workflows/ci.yml` — lint, typecheck, test, format,
  storybook build on every PR

**Button** ported from vanilla HTML to React with CSS Modules:
forwardRef, full props (variant × size × iconOnly × fullWidth ×
loading × leftIcon/rightIcon), 18 vitest tests including 3 axe a11y
assertions.

Distribution model decided: **three channels in parallel** — vendored
copy (existing path), npm package (new), copy-and-own CLI (Phase 2+).
The build script writes to all relevant locations from one source.

## Phase 2 — TextField + Dialog + Toast + scaffold CLI (PR #31)

Three more production-grade components:

- **TextField**: label / helperText / error / labelAction / multiline,
  auto-wired `aria-describedby`, `role="alert"` on error message,
  `useId()` linkage. 20 tests.
- **Dialog**: composition API (`Dialog.Title` / `.Description` /
  `.Body` / `.Actions` / `.ActionPrimary` / `.ActionSecondary`),
  self-contained portal + focus trap + Escape close + scroll lock +
  return-focus, **no Radix dependency**. Auto-detects `Dialog.Title`
  via `Children.forEach` for synchronous `aria-labelledby` wiring.
  13 tests including focus-trap.
- **Toast**: `<ToastProvider>` + `useToast()` imperative API. Variants
  (positive / warn / negative / normal) map to function tokens.
  Auto-dismiss with hover-pause; dedupe by id; queue limit. 13 tests.

**scaffold-component CLI** (`scripts/scaffold-component.mjs`)
implemented from its Phase 1 stub: `node scripts/scaffold-component.mjs
Button --to ./src/components --name MyButton` copies sources into a
consumer repo with optional symbol rename.

`docs/consume.md` restructured around a decision tree at the top:
"Static landing → vendored copy / React app → npm package /
Fork-and-own → scaffold CLI".

## Phase 3 — Storybook deploy + tokens docs site + freewrite isolation (PR #32)

Public-facing docs went live at
**https://thematters.github.io/design-system/** via GitHub Actions
(`actions/configure-pages` + `upload-pages-artifact` + `deploy-pages`).
The Storybook now serves both component stories AND tokens reference
pages.

Six MDX docs pages under `apps/storybook/src/tokens/`:

- **Brand** — canonical New Palette + deprecated legacy palette, with
  the "Logo/Matters Green → `--color-brand-new-purple`" remap rule
- **Colors** — Primary / Secondary / Grey / Function / Info /
  Attention scales, all chips read live from `@matters/design-system-tokens`
- **Typography** — System + Article scales rendered as live samples
- **Spacing** — table with visual purple bars at each token width
- **Shadows** — four elevation cards
- **Freewrite** — 七日書 opt-in palette + import snippet

**Freewrite isolation** (mechanical enforcement): the build script
strips `--color-freewrite-*` from the main `tokens.css` and emits them
to a separate `freewrite.css`. Non-Freewrite product code that
references those vars now sees them undefined — that's the intended
failure mode.

## Phase 4 — Avatar + Banner + ArticleCard + Figma Code Connect (PR #34, #40)

Three compound components closer to product surfaces:

- **Avatar** — 4 sizes (xs/sm/md/lg) × 4 ring variants (none /
  architect / civicLiker / both) × badge slot. Initials fallback for
  missing-src and image-load-error. Smart CJK initials (last 2 chars).
- **Banner** — single carousel slide; full-bleed image + dark scrim +
  2-line title clamp + optional pagination dots + optional `<a>`
  wrapper. The carousel container is the consumer's choice.
- **ArticleCard** — feed card. Author + time head row, title +
  summary + cover image content row, optional `meta` slot + more-actions
  footer row. Skeleton placeholder state. Both Small (76px cover) and
  Big (106px cover). `meta` is a free-form `ReactNode` slot rather than
  baked-in 編輯選擇 / 選集 / 七日書 / 圍爐 props — those are business
  semantics that change over time and don't belong in the DS API.

**Figma Code Connect** wired for Button / Avatar / Banner / ArticleCard
in PR #34 and TextField / Dialog / Toast in PR #40. `<Component>.figma.tsx`
files map Figma variant properties to React props; designers using Dev
Mode see the live React snippet next to the component instance.
Manual-trigger publish workflow at `.github/workflows/figma-code-connect.yml`
(needs `FIGMA_ACCESS_TOKEN` secret on first run).

Components total: **97/97 tests pass, 11 axe a11y assertions** across
the seven shipped components.

## Phase 5 — changesets + npm publish + Playwright visual regression (PR #35, plus #36 and #37 fixes)

**Versioning**: changesets (`@changesets/cli`) drives the release
workflow. `tokens` and `react` are linked at major.minor. Per-PR
contributors run `pnpm changeset` to declare a bump; merging the
auto-opened "Version Packages" PR triggers npm publish. Documented in
`docs/releasing.md`.

**Release CI** (`.github/workflows/release.yml`): on push to main,
runs `pnpm run version` (changesets bumps + writes CHANGELOG.md, opens
PR) and `pnpm run release` (`scripts/release.sh` builds packages and
runs `changeset publish` if `NPM_TOKEN` is set; gracefully skips with
`::warning::` if not).

PR #36 fixed the GitHub-info changelog plugin failure ("escaped
applying the changesets"). PR #37 fixed the bare `pnpm version`
shadowing pnpm's built-in version subcommand — now uses `pnpm run
version` explicitly.

**Visual regression** (`playwright.config.ts` +
`tests/visual/components.spec.ts`): Chromium against the built
Storybook, 17 stories snapshotted, 1% diff tolerance, animations
disabled. Linux-pinned baseline; `workflow_dispatch update=true`
regenerates and commits. Documented in `docs/visual-regression.md`.

**First Version PR** opened automatically (#38) and merged →
`@matters/design-system-tokens@0.2.0` and
`@matters/design-system-react@0.2.0` are version-tagged in git.
Actual npm publish pending `NPM_TOKEN` secret (org access being
arranged).

## Phase 6 — Operational templates (PR #39)

The "promotion image" half of the original four delivery buckets.
Three HTML+CSS templates rendered to deterministic PNG via Playwright,
no Figma seat required:

- **OG image** 1200×630 — article share card (`og:image`,
  `twitter:image`)
- **Social card** 1080×1080 — quote-forward share for Threads / IG /
  Mastodon
- **Newsletter header** 600×200 — email banner

`scripts/render-template.mjs` opens the template, substitutes
`{{key}}` placeholders from JSON data, sets the viewport per the
template's `<meta name="ds-template-size">` tag, waits for fonts +
images, screenshots `.tpl-canvas`. ~1s per render on M1, ~2s in CI.

`pnpm template:render <name> --data <json> --out <png>` shorthand.

Why HTML+Playwright over Satori or Figma export: text-editable by
ops, version-controlled, tokens stay automatically in sync with the
React package + Storybook, and 100 article OGs is just a shell loop.
Noto Sans/Serif TC via Google Fonts so output is identical across
macOS / Linux / CI runners.

Documented in `docs/templates.md` including "When the data is
dynamic" — pointer toward an HTTP wrapper service when matters.town
backend wants auto-OG generation at write time.

## Cheat sheet

| Want to… | Do this |
|---|---|
| Change a token value | edit `packages/tokens/tokens.json` → `pnpm build:tokens` |
| Add a new React component | `packages/react/src/components/<Name>/` + `components/<group>/<name>/spec.md` (sister files) |
| Edit Storybook | `apps/storybook/.storybook/` or any component's `*.stories.tsx` |
| Add a Figma↔code mapping | `<Component>.figma.tsx` next to the component; see `docs/code-connect.md` |
| Render an OG image | `pnpm template:render og-image --data my.json --out my.png` |
| Cut a release | `pnpm changeset` per PR; merge the auto-opened Version PR; `NPM_TOKEN` does the publish |
| Update visual regression baseline | Actions → Visual regression → Run workflow → `update=true` |
| See what shipped when | `packages/<name>/CHANGELOG.md` (auto), GitHub Releases, this file |

## How long it took

- **Day 1 (2026-04-24)**: Phase 0
- **Day 2 (2026-04-25)**: Phases 1 → 5, plus parallel refresh of 3 consumer repos overnight
- **Day 3 (2026-04-26)**: Phase 6, Phase 4 follow-up (Code Connect for Phase 2 components), Phase 7 in flight (render service), and the ArticleCard cover-overhang fix that you (the reader, presumably) caught in Storybook autodocs

Most of the pace came from delegating consumer-repo refresh work to
parallel agents while the main repo was at a settled point — design
system PRs in foreground, downstream PRs in background. The choice to
land Phase 5 rails (changesets + visual regression + auto-publish)
**before** doing the heaviest implementation work paid back: every
subsequent PR landed on a stable foundation.
